아래 "커밋 루프" 규칙으로 작업한다. 작업 대상: $ARGUMENTS

# 0. 분해 (한 번만)
- 전체 작업을 순서가 있는 "커밋 단위" 목록으로 쪼갠다. 각 커밋은 하나의 논리적 변경(독립적으로 리뷰/되돌리기 가능한 단위)이어야 한다. plan이 크더라도 반드시 커밋 사이즈로 분해한다.
- 분해안을 제시할 때 **셀프 리뷰**도 함께 붙인다: 리스크·버린 대안·빠졌을 수 있는 부분을 짧게 짚어 승인이 더 정확해지게 한다. **큰/아키텍처성 작업**이면 구현 전 접근(분해안)을 한 번 더 검증한다(필요하면 별도 리뷰 — '다른 눈').
- 이 분해 목록(+셀프 리뷰)만 먼저 제시하고 **멈춰서 내 승인을 기다린다.** 내가 순서·범위 수정을 요청할 수 있다.

# 1..N. 각 커밋 단위 루프
승인되면 첫 커밋부터 다음을 반복한다:

1. **커밋 plan**: 이번 커밋에서 건드릴 파일과 변경 내용을 간단히 제시하고 **멈춰서 내 승인을 기다린다.** (일반 plan과 동일한 interactive 게이트. 내가 수정 요청 가능.)
2. **구현**: 승인된 범위만 구현한다. 범위를 벗어나지 않는다.
3. **빌드/테스트**: 이 repo의 빌드/테스트를 실행한다. 명령은 repo의 `.githooks/pre-commit`(또는 repo CLAUDE.md에 명시된 빌드 커맨드)을 **단일 소스**로 따른다. 모르면 나에게 물어본다. 실패하면 통과할 때까지 **스스로 고친다.** 통과 전에는 리뷰/확인으로 넘어가지 않는다.
4. **리뷰**: `code-reviewer` 서브에이전트를 호출해 변경분을 리뷰받고, 결과를 [Critical]/[Warning]/[Suggestion]으로 정리해 보여준다.
5. **확인 게이트**: 여기서 **반드시 멈춰서 내 확인을 기다린다.** 내가:
   - 수정 요청 → 반영하고 2~4 반복
   - "커밋했어" / "다음" → 다음 커밋으로
6. **승인 없이 스스로 커밋하지 않는다.** 5의 확인 게이트에서 멈춰 내 OK를 받은 뒤에만 커밋한다(확인만 받았으면 커밋은 네가 해도, 내가 해도 된다). 절대 스스로 다음 커밋으로 진행하지 않는다. 한 텀(커밋 단위)이 끝나면 멈춘다.

# 예외
- 단순 질문/설명/조사 요청, 그리고 **아주 사소한 한두 줄 수정**(오타·상수·로그 등)에는 이 루프를 적용하지 않는다. 빌드/테스트/리뷰 없이 바로 처리한다.
